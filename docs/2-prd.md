## 1. 제품 개요

| 항목 | 내용 |
|------|------|
| 제품명 | TodoListApp |
| 목적 | 인증된 사용자 단위로 개인 할일을 카테고리별로 관리하는 반응형 웹 애플리케이션 |
| 플랫폼 | Web / Mobile Web (반응형 웹 UI) |
| 개발 기간 | 3일 (MVP 1차 릴리즈 기준) |

### 해결하는 문제

1. 개인별 할일 데이터를 서버 DB에 안전하게 저장·관리하는 수단 부재
2. 인증된 사용자 단위 데이터 격리를 통한 프라이버시 보장 미흡
3. 사용자 정의 카테고리를 통한 유연한 할일 분류 기능 부재

## 3. 기술 스택

| 영역 | 기술 |
|------|------|
| 프론트엔드 | React 19 + TypeScript |
| 상태 관리 | Zustand |
| 서버 상태 / 캐싱 | TanStack Query |
| 백엔드 | Node.js + Express (REST API) |
| 데이터베이스 | PostgreSQL 17 |
| DB 접근 라이브러리 | **pg** (node-postgres) — 다른 ORM/쿼리 빌더 사용 금지 |
| 인증 | JWT Access Token + Refresh Token (1차), OAuth Social(Google, Facebook 등) 확장 예정 (2차) |
| 프론트엔드 빌드 | Vite |
| 테스트 (FE) | Vitest + React Testing Library |
| 테스트 (BE) | Jest + Supertest |

### 아키텍처 원칙

- 백엔드는 Stateless REST API 서버로 운영한다.
- 모든 DB 쿼리는 `pg` 라이브러리를 직접 사용하여 작성한다.
- JWT는 Access Token + Refresh Token 이중 구조로 발급한다.
  - Access Token: 만료 1시간, `Authorization: Bearer` 헤더로 전달
  - Refresh Token: 만료 7일, Zustand 메모리에 저장 (페이지 새로고침 시 소멸)
  - `POST /api/auth/refresh` 엔드포인트로 Access Token 재발급, Refresh Token은 요청 바디로 전달
- 향후 OAuth 확장을 고려한 인증 레이어 추상화를 적용한다.

## 5. 기능 요구사항

### 5.1 MVP 범위 (1차 릴리즈 — 3일 내 완료)

도메인 정의서의 유스케이스 전체를 1차 MVP에 포함하며, 아래 항목을 추가한다.

#### 인증 (Identity & Access)

| UC | 기능 | 상세 요구사항 |
|----|------|---------------|
| UC-01 | 회원가입 | 이메일 + 비밀번호 입력, 이메일 중복 검증, 비밀번호 bcrypt 암호화, 가입 후 즉시 JWT 발급 |
| UC-02 | 로그인 | 이메일 + 비밀번호 검증, 성공 시 Access Token + Refresh Token 발급, 실패 시 명확한 오류 메시지 |
| UC-02a | 로그아웃 | 서버에서 Refresh Token 무효화, 클라이언트 Zustand의 Access Token·Refresh Token·사용자 정보 초기화 |
| UC-02b | 토큰 재발급 | Refresh Token 유효 시 새 Access Token 발급, 만료 시 로그인 페이지로 리다이렉트 |
| UC-03 | 개인정보 수정 | 표시 이름 수정 및 비밀번호 변경 가능, 비밀번호 변경 시 현재 비밀번호 확인 필수, 본인 정보만 수정 가능 |
| UC-12 | 회원 탈퇴 | 본인 확인(비밀번호 재입력) 후 계정 삭제, 해당 사용자의 할일·커스텀 카테고리 즉시 삭제 (CASCADE) |

#### 카테고리 관리 (Todo Management)

| UC | 기능 | 상세 요구사항 |
|----|------|---------------|
| UC-04 | 기본 카테고리 조회 | 전체 사용자 공통 제공, 수정·삭제 불가 |
| UC-05 | 사용자 정의 카테고리 생성 | 이름 필수, 생성 즉시 할일 등록 시 선택 가능 |
| UC-06 | 사용자 정의 카테고리 수정·삭제 | 본인 생성 카테고리만 대상, 기본 카테고리 변경 불가. **삭제 시 해당 카테고리 소속 할일은 시스템 기본 카테고리(첫 번째 기본 카테고리)로 자동 이동** |

#### 할일 관리 (Todo Management)

| UC | 기능 | 상세 요구사항 |
|----|------|---------------|
| UC-07 | 할일 등록 | 제목 필수 / 설명·종료예정일·카테고리 선택, 카테고리 미선택 시 기본 카테고리 자동 지정 |
| UC-08 | 할일 목록 조회 및 필터링 | 카테고리·기간(종료예정일 범위)·완료 여부 단독 및 복합 필터, 필터 미적용 시 전체 반환 |
| UC-09 | 할일 수정 | 본인 할일만 수정 가능, 수정 후 `updated_at` 갱신 |
| UC-10 | 완료 상태 토글 | 완료 ↔ 미완료 전환, 데이터 삭제 없이 상태만 변경 |
| UC-11 | 할일 삭제 | 본인 할일만 삭제 가능, 삭제 후 목록에서 즉시 제거 |

## 6. API 설계 방향

RESTful 설계 원칙을 따르며 아래 엔드포인트 그룹으로 구성한다.

| 그룹 | 기본 경로 | 주요 작업 |
|------|-----------|-----------|
| 인증 | `/api/auth` | 회원가입, 로그인, 로그아웃, 토큰 재발급 |
| 사용자 | `/api/users/me` | 개인정보 조회·수정(이름·비밀번호), 회원 탈퇴 |
| 카테고리 | `/api/categories` | 목록 조회, 생성, 수정, 삭제 |
| 할일 | `/api/todos` | 목록 조회(필터), 생성, 수정, 삭제, 완료 토글 |

- `/api/users`, `/api/categories`, `/api/todos` 요청은 `Authorization: Bearer <Access Token>` 헤더 필수
- `/api/auth/refresh`는 요청 바디의 Refresh Token으로 인증
- 오류 응답 형식은 `{ "error": { "code": "...", "message": "..." } }` 로 통일

## 7. 데이터 모델 요약

도메인 정의서(`1-domain-definition.md`) §3을 기준으로 하며, PostgreSQL DDL 작성 시 아래 원칙을 적용한다.

- PK는 `UUID` 타입 (`gen_random_uuid()` 기본값)
- 타임스탬프는 `TIMESTAMPTZ` 사용
- `todos.user_id` → `users.id` ON DELETE CASCADE
- `todos.category_id` → `categories.id` ON DELETE 시 기본 카테고리 ID로 SET DEFAULT (UC-06 정책 반영)
- `categories.user_id` → `users.id` ON DELETE CASCADE (사용자 정의 카테고리)
- Refresh Token은 `refresh_tokens` 테이블에 저장 (`user_id`, `token_hash`, `expires_at`, `revoked`)

## 9. 보안 요구사항

| 항목 | 요구사항 |
|------|---------|
| 비밀번호 | bcrypt 해시 저장 (salt rounds ≥ 10) |
| 비밀번호 변경 | 현재 비밀번호 확인 후 변경, 변경 즉시 기존 Refresh Token 전체 무효화 |
| JWT Access Token | 서버 서명 검증, 만료 1시간 |
| JWT Refresh Token | 만료 7일, Zustand 메모리 저장 (페이지 새로고침 시 소멸), 로그아웃·비밀번호 변경 시 서버 DB에서 즉시 무효화 |
| 데이터 격리 | 모든 조회·수정·삭제 쿼리에 `user_id` 조건 포함 |
| 입력 검증 | 서버 사이드에서 모든 요청 바디 유효성 검증 |
| SQL Injection | `pg` 라이브러리의 파라미터 바인딩(`$1, $2, ...`) 방식 사용, 문자열 직접 삽입 금지 |

## 10. 비즈니스 규칙 요약

도메인 정의서 §4의 BR-01 ~ BR-11 전체 적용. 핵심 항목:

- **BR-01/02**: 미인증 접근 차단, 본인 데이터만 접근 가능
- **BR-03**: 이메일은 전체 시스템에서 유일, 로그인 식별자로 사용
- **BR-04**: 기본 카테고리 수정·삭제 불가
- **BR-05**: 사용자 정의 카테고리는 해당 사용자만 조회·수정·삭제 가능
- **BR-06**: 할일은 반드시 하나의 카테고리에 속해야 함 (카테고리 삭제 시 기본 카테고리로 자동 이동으로 충족)
- **BR-07**: 할일 제목 필수
- **BR-08**: 종료예정일은 과거 날짜 허용
- **BR-09**: 완료 처리는 삭제가 아닌 상태 변경으로 처리
- **BR-10**: 카테고리·기간·완료 여부 필터를 단독 또는 복합 적용 가능
- **BR-11**: 기간 필터 적용 시 종료예정일 없는 할일 제외
