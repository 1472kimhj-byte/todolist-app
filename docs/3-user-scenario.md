## UC-01. 회원가입

**액터:** 비인증 사용자
**목표:** 이메일과 비밀번호로 계정을 생성하고 서비스를 즉시 이용한다.

### 사전 조건

- 사용자가 회원가입 화면에 접근한 상태다.
- 사용자는 아직 계정이 없다.

### 기본 흐름

1. 사용자가 이메일 주소와 비밀번호를 입력 폼에 작성한다.
2. 사용자가 '회원가입' 버튼을 누른다.
3. 시스템이 입력된 이메일이 이미 사용 중인지 확인한다. [BR-03]
4. 시스템이 비밀번호를 암호화하여 저장한다.
5. 시스템이 신규 계정을 생성하고 즉시 Access Token과 Refresh Token을 발급한다.
6. 사용자가 로그인된 상태로 할일 목록 화면으로 이동한다.

### 예외 흐름

- **E1. 이메일 중복:** 3단계에서 동일한 이메일이 이미 존재할 경우 [BR-03], 시스템이 "이미 사용 중인 이메일입니다" 메시지를 표시하고 흐름을 중단한다.
- **E2. 필수 항목 누락:** 이메일 또는 비밀번호를 입력하지 않은 경우, 시스템이 해당 항목의 오류 메시지를 표시하고 제출을 차단한다.
- **E3. 이메일 형식 오류:** 유효하지 않은 이메일 형식을 입력한 경우, 시스템이 형식 오류 메시지를 표시한다.

### 사후 조건

- 새 계정이 데이터베이스에 저장된다.
- 사용자는 인증된 상태로 할일 목록 화면에 진입한다.
- Access Token과 Refresh Token이 Zustand 메모리에 저장된다.

## UC-02. 로그인

**액터:** 비인증 사용자
**목표:** 이메일과 비밀번호로 본인 계정에 접속하여 할일을 관리한다.

### 사전 조건

- 사용자가 로그인 화면에 접근한 상태다.
- 사용자가 이미 계정을 보유하고 있다.

### 기본 흐름

1. 사용자가 이메일 주소와 비밀번호를 입력 폼에 작성한다.
2. 사용자가 '로그인' 버튼을 누른다.
3. 시스템이 입력된 이메일로 계정을 조회한다. [BR-03]
4. 시스템이 입력된 비밀번호를 저장된 암호화 비밀번호와 대조한다.
5. 검증이 성공하면 시스템이 Access Token과 Refresh Token을 발급한다.
6. 사용자가 할일 목록 화면으로 이동한다.

### 예외 흐름

- **E1. 이메일 미존재:** 3단계에서 해당 이메일로 등록된 계정이 없을 경우, 시스템이 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지를 표시하고 흐름을 중단한다.
- **E2. 비밀번호 불일치:** 4단계에서 비밀번호가 맞지 않을 경우, 시스템이 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지를 표시하고 흐름을 중단한다.
- **E3. 필수 항목 누락:** 이메일 또는 비밀번호를 입력하지 않은 경우, 시스템이 해당 항목의 오류 메시지를 표시하고 제출을 차단한다.

### 사후 조건

- 사용자는 인증된 상태로 할일 목록 화면에 진입한다.
- Access Token과 Refresh Token이 Zustand 메모리에 저장된다.

## UC-02a. 로그아웃

**액터:** 인증된 사용자
**목표:** 현재 세션을 안전하게 종료하고 인증 상태를 초기화한다.

### 사전 조건

- 사용자가 로그인된 상태다.
- 유효한 Access Token과 Refresh Token이 존재한다.

### 기본 흐름

1. 사용자가 '로그아웃' 버튼을 누른다.
2. 시스템이 서버에 로그아웃 요청을 전송한다.
3. 서버가 해당 Refresh Token을 즉시 무효화한다.
4. 시스템이 Zustand 메모리의 Access Token·Refresh Token·사용자 정보를 초기화한다.
6. 사용자가 로그인 화면으로 이동한다.

### 예외 흐름

- **E1. 서버 통신 실패:** 2단계에서 서버 응답이 없을 경우, 시스템은 Zustand 인증 상태를 강제로 초기화하고 로그인 화면으로 이동한다.

### 사후 조건

- 서버의 Refresh Token이 무효화된다.
- 클라이언트에 인증 정보가 남아 있지 않다.
- 사용자는 비인증 상태로 전환된다.

## UC-02b. 토큰 재발급

**액터:** 인증된 사용자
**목표:** Access Token 만료 시 자동으로 새 토큰을 발급받아 서비스 이용을 지속한다.

### 사전 조건

- 사용자가 로그인된 상태에서 서비스를 이용 중이다.
- Access Token이 만료되었다.
- 유효한 Refresh Token이 Zustand 메모리에 존재한다.

### 기본 흐름

1. 사용자가 화면에서 특정 기능(할일 조회, 등록 등)을 수행한다.
2. 시스템이 API 요청을 전송하고 서버로부터 Access Token 만료 응답을 받는다.
3. 시스템이 Refresh Token을 사용하여 자동으로 토큰 재발급을 요청한다.
4. 서버가 Refresh Token의 유효성을 확인하고 새 Access Token을 발급한다.
5. 시스템이 새 Access Token을 전역 인증 상태(Zustand)에 저장한다.
6. 시스템이 원래 요청을 재시도하고 정상 결과를 화면에 반영한다.

### 예외 흐름

- **E1. Refresh Token 만료:** 4단계에서 Refresh Token도 만료된 경우, 시스템이 인증 상태를 초기화하고 사용자를 로그인 화면으로 이동시킨다.
- **E2. Refresh Token 무효화:** 로그아웃 또는 비밀번호 변경으로 Refresh Token이 서버에서 무효화된 경우, 시스템이 인증 상태를 초기화하고 로그인 화면으로 이동시킨다.

### 사후 조건

- 새 Access Token이 발급되어 전역 상태에 저장된다.
- 사용자는 세션 중단 없이 서비스를 계속 이용한다.

## UC-03. 개인정보 수정

**액터:** 인증된 사용자
**목표:** 표시 이름 또는 비밀번호를 변경하여 계정 정보를 최신 상태로 유지한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 사용자가 개인정보 수정 화면에 접근한 상태다.

### 기본 흐름 — 표시 이름 수정

1. 사용자가 표시 이름 입력란에 새 이름을 작성한다.
2. 사용자가 '저장' 버튼을 누른다.
3. 시스템이 본인의 계정 정보인지 확인한다. [BR-02]
4. 시스템이 새 이름을 저장하고 updated_at을 갱신한다.
5. 화면에 변경된 이름이 즉시 반영된다.

### 기본 흐름 — 비밀번호 변경

1. 사용자가 현재 비밀번호, 새 비밀번호를 각각 입력란에 작성한다.
2. 사용자가 '비밀번호 변경' 버튼을 누른다.
3. 시스템이 입력된 현재 비밀번호가 일치하는지 확인한다.
4. 시스템이 새 비밀번호를 암호화하여 저장하고 updated_at을 갱신한다.
5. 시스템이 해당 사용자의 모든 기존 Refresh Token을 서버에서 즉시 무효화한다.
6. 사용자에게 변경 완료 안내 메시지가 표시된다.

### 예외 흐름

- **E1. 현재 비밀번호 불일치:** 비밀번호 변경 흐름의 3단계에서 현재 비밀번호가 맞지 않을 경우, 시스템이 "현재 비밀번호가 올바르지 않습니다" 메시지를 표시하고 흐름을 중단한다.
- **E2. 타인 계정 수정 시도:** 3단계에서 본인 계정이 아닌 경우 [BR-02], 시스템이 권한 오류를 반환하고 요청을 거부한다.
- **E3. 이름 입력값 누락:** 표시 이름이 비어 있는 경우, 시스템이 필수 입력 오류 메시지를 표시한다.

### 사후 조건

- 변경된 표시 이름 또는 비밀번호가 데이터베이스에 저장된다.
- 비밀번호 변경 시 기존 Refresh Token이 모두 무효화된다.

## UC-04. 기본 카테고리 조회

**액터:** 인증된 사용자
**목표:** 시스템이 제공하는 기본 카테고리 목록을 확인하여 할일 분류 기준으로 활용한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 할일 목록 화면 또는 할일 등록 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 카테고리 목록이 표시되는 화면에 진입한다.
2. 시스템이 전체 사용자 공통으로 제공되는 기본 카테고리 목록을 불러온다. [BR-04]
3. 화면에 기본 카테고리 목록이 표시된다.
4. 사용자가 기본 카테고리 항목을 확인한다.

### 예외 흐름

- **E1. 인증 만료:** 시스템이 인증되지 않은 요청으로 판단하면 [BR-01], 사용자를 로그인 화면으로 이동시킨다.

### 사후 조건

- 기본 카테고리 목록이 화면에 표시된다.
- 기본 카테고리는 수정 또는 삭제 버튼이 제공되지 않는다. [BR-04]

## UC-05. 사용자 정의 카테고리 생성

**액터:** 인증된 사용자
**목표:** 개인 맞춤 카테고리를 새로 만들어 할일을 원하는 방식으로 분류한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 사용자가 카테고리 관리 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 '카테고리 추가' 버튼을 누른다.
2. 사용자가 새 카테고리 이름을 입력란에 작성한다.
3. 사용자가 '저장' 버튼을 누른다.
4. 시스템이 해당 사용자의 소유로 새 카테고리를 저장한다. [BR-05]
5. 새 카테고리가 카테고리 목록에 즉시 추가된다.
6. 이후 할일 등록 시 새 카테고리를 선택할 수 있다.

### 예외 흐름

- **E1. 카테고리 이름 누락:** 3단계에서 이름을 입력하지 않은 경우, 시스템이 "카테고리 이름을 입력해주세요" 메시지를 표시하고 저장을 차단한다.

### 사후 조건

- 새 카테고리가 해당 사용자 소유로 데이터베이스에 저장된다. [BR-05]
- 할일 등록·수정 화면의 카테고리 선택 목록에 즉시 반영된다.

---

## UC-06. 사용자 정의 카테고리 수정·삭제

**액터:** 인증된 사용자
**목표:** 본인이 만든 카테고리의 이름을 변경하거나 더 이상 필요 없는 카테고리를 삭제한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 수정 또는 삭제 대상 카테고리가 본인이 생성한 사용자 정의 카테고리다. [BR-02] [BR-05]
- 사용자가 카테고리 관리 화면에 접근한 상태다.

### 기본 흐름 — 수정

1. 사용자가 본인 소유의 카테고리 옆 '수정' 버튼을 누른다.
2. 사용자가 카테고리 이름 입력란에서 새 이름을 작성한다.
3. 사용자가 '저장' 버튼을 누른다.
4. 시스템이 해당 카테고리가 본인 소유인지 확인한다. [BR-02] [BR-05]
5. 시스템이 변경된 이름을 저장하고 목록을 갱신한다.

### 기본 흐름 — 삭제

1. 사용자가 본인 소유의 카테고리 옆 '삭제' 버튼을 누른다.
2. 시스템이 삭제 확인 안내 메시지를 표시한다.
3. 사용자가 삭제를 최종 확인한다.
4. 시스템이 해당 카테고리가 본인 소유인지 확인한다. [BR-02] [BR-05]
5. 시스템이 해당 카테고리에 속한 모든 할일을 첫 번째 기본 카테고리로 자동 이동한다. [BR-06]
6. 시스템이 해당 카테고리를 삭제하고 목록을 갱신한다.

### 예외 흐름

- **E1. 기본 카테고리 수정·삭제 시도:** 사용자가 기본 카테고리에서 수정 또는 삭제를 시도할 경우 [BR-04], 해당 버튼이 비활성화되어 있거나 시스템이 "기본 카테고리는 변경할 수 없습니다" 메시지를 표시하고 요청을 거부한다.
- **E2. 타인 카테고리 접근 시도:** 4단계에서 본인 소유가 아닌 카테고리인 경우 [BR-02] [BR-05], 시스템이 권한 오류를 반환하고 요청을 거부한다.
- **E3. 카테고리 이름 누락 (수정 시):** 새 이름을 입력하지 않은 경우, 시스템이 오류 메시지를 표시하고 저장을 차단한다.

### 사후 조건

- (수정) 카테고리 이름이 변경되어 데이터베이스에 저장된다.
- (삭제) 카테고리가 삭제되고, 해당 카테고리에 속했던 할일은 첫 번째 기본 카테고리로 이동한다. [BR-06]

## UC-07. 할일 등록

**액터:** 인증된 사용자
**목표:** 새 할일을 등록하여 관리 목록에 추가한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 사용자가 할일 등록 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 할일 제목을 입력란에 작성한다. [BR-07]
2. (선택) 사용자가 설명, 종료예정일, 카테고리를 입력하거나 선택한다. [BR-08]
3. 사용자가 '저장' 버튼을 누른다.
4. 시스템이 카테고리 선택 여부를 확인한다.
   - 카테고리를 선택하지 않은 경우: 시스템이 첫 번째 기본 카테고리를 자동으로 지정한다. [BR-06]
5. 시스템이 새 할일을 해당 사용자 소유로 저장한다. [BR-02]
6. 새 할일이 할일 목록에 즉시 추가되어 화면에 반영된다.

### 예외 흐름

- **E1. 제목 누락:** 제목을 입력하지 않고 저장을 시도하면 [BR-07], 시스템이 "할일 제목을 입력해주세요" 메시지를 표시하고 저장을 차단한다.

### 사후 조건

- 새 할일이 해당 사용자 소유로 데이터베이스에 저장된다.
- 할일은 반드시 하나의 카테고리에 속한 상태로 저장된다. [BR-06]
- 할일이 할일 목록 화면에 즉시 표시된다.

## UC-08. 할일 목록 조회 및 필터링

**액터:** 인증된 사용자
**목표:** 등록된 할일 목록을 조회하고 필요에 따라 조건을 적용하여 원하는 할일만 추려서 본다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 사용자가 할일 목록 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 할일 목록 화면에 진입한다.
2. 시스템이 해당 사용자 소유의 모든 할일을 불러온다. [BR-02]
3. 할일 목록이 화면에 표시된다.
4. (선택) 사용자가 카테고리, 기간(종료예정일 범위), 완료 여부 중 하나 이상의 필터를 선택한다. [BR-10]
5. 시스템이 선택된 필터 조건에 맞는 할일만 추려서 반환한다.
   - 기간 필터 적용 시: 종료예정일이 없는 할일은 결과에서 제외된다. [BR-11]
6. 필터링된 결과가 화면에 즉시 반영된다.

### 예외 흐름

- **E1. 필터 결과 없음:** 선택한 조건에 해당하는 할일이 없을 경우, 시스템이 "조건에 맞는 할일이 없습니다" 안내 메시지를 표시한다.
- **E2. 필터 미적용:** 필터를 선택하지 않은 경우, 시스템이 해당 사용자의 전체 할일을 반환한다. [BR-10]

### 사후 조건

- 조건에 맞는 할일 목록이 화면에 표시된다.
- 필터 조건이 화면 UI에 선택된 상태로 유지된다.

## UC-09. 할일 수정

**액터:** 인증된 사용자
**목표:** 이미 등록한 할일의 내용을 변경하여 최신 상태로 유지한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 수정 대상 할일이 본인 소유의 할일이다. [BR-02]
- 사용자가 해당 할일의 수정 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 수정할 할일 항목의 '수정' 버튼을 누른다.
2. 현재 할일의 내용(제목, 설명, 종료예정일, 카테고리)이 수정 폼에 표시된다.
3. 사용자가 변경할 항목을 수정한다.
4. 사용자가 '저장' 버튼을 누른다.
5. 시스템이 해당 할일이 본인 소유인지 확인한다. [BR-02]
6. 시스템이 변경된 내용을 저장하고 updated_at을 현재 시각으로 갱신한다.
7. 할일 목록 화면에 수정된 내용이 즉시 반영된다.

### 예외 흐름

- **E1. 제목 삭제:** 4단계에서 제목을 지우고 저장을 시도하면 [BR-07], 시스템이 "할일 제목을 입력해주세요" 메시지를 표시하고 저장을 차단한다.
- **E2. 타인 할일 수정 시도:** 5단계에서 본인 소유가 아닌 경우 [BR-02], 시스템이 권한 오류를 반환하고 요청을 거부한다.

### 사후 조건

- 변경된 할일 내용이 데이터베이스에 저장된다.
- updated_at이 수정 시각으로 갱신된다.
- 할일 목록 화면에 수정 내용이 즉시 반영된다.

## UC-10. 완료 상태 토글

**액터:** 인증된 사용자
**목표:** 할일의 완료 여부를 전환하여 진행 상황을 기록한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 대상 할일이 본인 소유의 할일이다. [BR-02]
- 사용자가 할일 목록 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 할일 항목의 완료 체크박스 또는 완료 버튼을 누른다.
2. 시스템이 해당 할일이 본인 소유인지 확인한다. [BR-02]
3. 시스템이 해당 할일의 완료 여부를 반대 상태로 전환한다. [BR-09]
   - 미완료(Pending) 상태인 경우: 완료(Done)로 변경한다.
   - 완료(Done) 상태인 경우: 미완료(Pending)로 되돌린다.
4. 할일 목록 화면에서 해당 항목의 상태가 즉시 변경된다.

### 예외 흐름

- **E1. 타인 할일 상태 변경 시도:** 2단계에서 본인 소유가 아닌 경우 [BR-02], 시스템이 권한 오류를 반환하고 요청을 거부한다.

### 사후 조건

- 할일의 완료 여부(is_completed) 값이 데이터베이스에서 변경된다. [BR-09]
- 할일 데이터는 삭제되지 않고 상태만 변경된다. [BR-09]
- 화면의 할일 항목 표시가 변경된 상태로 즉시 업데이트된다.

## UC-11. 할일 삭제

**액터:** 인증된 사용자
**목표:** 더 이상 필요 없는 할일을 목록에서 영구적으로 제거한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 삭제 대상 할일이 본인 소유의 할일이다. [BR-02]
- 사용자가 할일 목록 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 삭제할 할일 항목의 '삭제' 버튼을 누른다.
2. 시스템이 삭제 확인 안내 메시지를 표시한다.
3. 사용자가 삭제를 최종 확인한다.
4. 시스템이 해당 할일이 본인 소유인지 확인한다. [BR-02]
5. 시스템이 해당 할일을 데이터베이스에서 삭제한다.
6. 해당 할일이 목록에서 즉시 제거되고 화면이 갱신된다.

### 예외 흐름

- **E1. 삭제 취소:** 3단계에서 사용자가 취소를 선택하면, 아무런 변경 없이 이전 화면 상태를 유지한다.
- **E2. 타인 할일 삭제 시도:** 4단계에서 본인 소유가 아닌 경우 [BR-02], 시스템이 권한 오류를 반환하고 요청을 거부한다.

### 사후 조건

- 해당 할일이 데이터베이스에서 영구 삭제된다.
- 할일 목록 화면에서 해당 항목이 즉시 사라진다.

## UC-12. 회원 탈퇴

**액터:** 인증된 사용자
**목표:** 계정과 모든 개인 데이터를 영구적으로 삭제하고 서비스 이용을 종료한다.

### 사전 조건

- 사용자가 로그인된 상태다. [BR-01]
- 사용자가 회원 탈퇴 화면에 접근한 상태다.

### 기본 흐름

1. 사용자가 회원 탈퇴 메뉴로 진입한다.
2. 시스템이 탈퇴 전 본인 확인을 위해 비밀번호 입력을 요구한다.
3. 사용자가 현재 비밀번호를 입력한다.
4. 시스템이 입력된 비밀번호가 일치하는지 확인한다.
5. 사용자가 탈퇴 최종 확인 버튼을 누른다.
6. 시스템이 해당 사용자의 모든 할일을 즉시 삭제한다.
7. 시스템이 해당 사용자의 모든 사용자 정의 카테고리를 즉시 삭제한다.
8. 시스템이 해당 사용자의 계정을 삭제한다.
9. 시스템이 Zustand 메모리의 인증 상태를 초기화한다.
10. 사용자가 서비스 초기 화면(로그인 또는 회원가입 화면)으로 이동한다.

### 예외 흐름

- **E1. 비밀번호 불일치:** 4단계에서 비밀번호가 맞지 않을 경우, 시스템이 "비밀번호가 올바르지 않습니다" 메시지를 표시하고 탈퇴 절차를 중단한다.
- **E2. 탈퇴 취소:** 5단계에서 사용자가 취소를 선택하면, 아무런 변경 없이 이전 화면 상태를 유지한다.

### 사후 조건

- 해당 사용자의 계정, 할일, 사용자 정의 카테고리가 모두 데이터베이스에서 삭제된다.
- 클라이언트에 인증 정보가 남아 있지 않다.
- 사용자는 동일한 이메일로 신규 회원가입이 가능한 상태가 된다. [BR-03]
